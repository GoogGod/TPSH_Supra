core/settings/ — настройки разбиты на три файла. base.py — всё, что не зависит от окружения: INSTALLED_APPS, конфигурация DRF (DEFAULT_AUTHENTICATION_CLASSES, DEFAULT_PERMISSION_CLASSES), настройки JWT (ACCESS_TOKEN_LIFETIME, REFRESH_TOKEN_LIFETIME), CORS_ALLOWED_ORIGINS. dev.py наследует base.py, добавляет DEBUG=True, SQLite, открытый CORS. prod.py наследует base.py, подключает PostgreSQL через dj-database-url, включает SECURE_SSL_REDIRECT, SECURE_HSTS_SECONDS, whitenoise middleware. Выбор файла — по переменной окружения DJANGO_ENV (development / production).

users/ — модель User наследует AbstractUser, добавляет поле role (choices: employee, manager, admin) и ForeignKey на Venue. Это единственное место, где определяется пользователь — ссылки из других приложений через settings.AUTH_USER_MODEL. Вся логика создания пользователей, назначения ролей, фильтрации по объекту — в services.py. Права доступа (permissions.py) используются во view-классах всех приложений.

shifts/ — ядро системы. Четыре модели: Venue (объект/ресторан — название, адрес, таймзона), Shift (конкретная смена — дата, время начала/конца, venue FK, required_staff), ShiftAssignment (привязка сотрудника к смене — shift FK, employee FK, status: assigned/confirmed/cancelled), Availability (окно доступности сотрудника — employee FK, дата, время начала/конца). Сервисный слой в services/: scheduler.py принимает venue и диапазон дат, запрашивает прогноз из forecasting, получает матрицу доступности и формирует назначения; validators.py проверяет конфликты (сотрудник на двух сменах одновременно), минимальный отдых между сменами, лимит часов за неделю.

forecasting/ — аналитический модуль, изолированный от остальной логики. HistoricalLoad — фактические данные загрузки (venue, дата, кол-во гостей/заказов, признак события/праздника). Forecast — сгенерированный прогноз (venue, целевая дата, прогнозная загрузка, рекомендуемое кол-во персонала, доверительный интервал). ModelMetadata — параметры обученной модели (алгоритм, дата обучения, MAE, accuracy). Сервисы: collector.py собирает и нормализует данные, predictor.py реализует обучение (SARIMA через statsmodels, резерв — scikit-learn для ensemble-моделей), evaluator.py считает метрики и сигнализирует о деградации ниже порога 85%. Management-команды позволяют запускать обучение из cron.

common/ — не является Django-приложением, не имеет моделей. Содержит кастомные исключения для единообразной обработки ошибок в DRF (exception_handler), настройки пагинации и миксин TimestampMixin для автоматических полей created_at, updated_at в моделях.
