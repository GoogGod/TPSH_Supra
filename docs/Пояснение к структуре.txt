ПОЯСНЕНИЕ К СТРУКТУРЕ (актуально)

1. backend/users
- Регистрация, логин/логаут, профиль, список пользователей.
- Роли:
  - admin
  - manager
  - employee_pro
  - employee_noob
- Менеджеры/сотрудники видят пользователей своего venue + admin.

2. backend/forecasting
- Управляет ML-запусками и прогнозами.
- Ключевые сущности:
  - ForecastRun (история запусков + метрики)
  - HourlyForecast (почасовые прогнозы)
- Важные endpoint'ы:
  - /forecast/upload-data/
  - /forecast/run/
  - /forecast/runs/
  - /forecast/accuracy/

3. backend/shifts
- Управляет объектами (venues) и расписаниями.
- Ключевые сущности:
  - Venue
  - MonthlySchedule
  - WaiterSlot
  - ScheduleEntry
- Основная логика:
  - статус расписания (есть/нет/опубликовано)
  - генерация draft через ML pipeline
  - редактирование draft
  - публикация
  - claim/assign/unassign слотов

4. backend/user_notifications
- Уведомления в системе:
  - публикация расписания
  - ручные назначения
  - подтверждение/отклонение назначений
- API для списка, счетчика непрочитанных, mark read, confirm/reject.

5. ml_data
- Отдельный ML-модуль, который backend вызывает как сервис.
- main.py: процессинг, обучение, прогноз, (опционально) расписание.
- scheduler.py: построение графика смен.
- Папки данных:
  - data/raw
  - data/processed
  - data/predicted/runs/venue_<id>/run_<id>

6. front
- SPA клиент на Vue.
- Использует только backend API.
- Корневые SPA-роуты обслуживаются fallback в backend core/urls.py.

7. Почему так разделено
- ML и API разделены, чтобы:
  - легче сопровождать модель отдельно от REST,
  - изолировать артефакты по venue/run,
  - избежать конфликтов при параллельных генерациях.
