ЦЕПОЧКА ДАННЫХ (актуально)

1) RAW ДАННЫЕ
- Источник: файл от бизнеса (.csv/.xlsx)
- Endpoint: POST /api/v1/forecast/upload-data/
- Результат: сохранение в ml_data/data/raw/real_orders.csv|xlsx

2) ОБУЧЕНИЕ/ПРОГНОЗ (ML)
- Endpoint: POST /api/v1/forecast/run/
- Для Admin доступно:
  process_data -> train_model -> make_forecast -> evaluate
- Для Manager: только make_forecast на готовой модели

3) ПАПКИ АРТЕФАКТОВ ПРОГНОЗА
- На каждый запуск создается изолированная папка:
  ml_data/data/predicted/runs/venue_<venue_id>/run_<run_id>/
- Минимум артефактов:
  - forecast.csv
  - waiter_schedule.csv (если генерировалось расписание)

4) ЗАГРУЗКА ПРОГНОЗА В БД
- hourly forecast загружается в таблицу HourlyForecast
- Доступ к просмотру:
  - GET /api/v1/forecast/hourly/
  - GET /api/v1/forecast/daily/

5) ГЕНЕРАЦИЯ РАСПИСАНИЯ
- Endpoint: POST /api/v1/schedule/generate/
- Внутри:
  a) запуск ML прогноза
  b) запуск planner/scheduler
  c) parse waiter_schedule.csv -> MonthlySchedule/WaiterSlot/ScheduleEntry
- Результат: черновик расписания на указанный month/year

6) РЕДАКТИРОВАНИЕ И ПУБЛИКАЦИЯ
- PATCH /api/v1/schedule/monthly/{id}/entries/bulk-update/ (только draft)
- POST /api/v1/schedule/monthly/{id}/publish/

7) ПОТРЕБЛЕНИЕ ФРОНТОМ
- Статус наличия: GET /api/v1/schedule/status/
- Список: GET /api/v1/schedule/monthly/
- Детали: GET /api/v1/schedule/monthly/{id}/
- Действия сотрудника: claim/confirm/reject через slots/notifications endpoints

8) ВАЖНО ПО ИЗОЛЯЦИИ
- Расчеты одного venue не должны перезаписывать данные другого.
- Для этого используется run-структура по venue/run и привязка ForecastRun -> venue.
